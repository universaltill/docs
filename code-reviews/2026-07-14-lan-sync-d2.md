## What shipped

- **`GET /api/sync/snapshot`** (bearer-gated, middleware-exempt): serves a
  `VACUUM INTO` snapshot — the backup mechanism doing double duty; safe
  while the primary is selling.
- **Join = enrol + snapshot + staged identity** (`joinPrimary`): after
  D1's enrolment the replica downloads the snapshot with its new bearer,
  stages it as the pending restore (P0.2's machinery verbatim), and
  writes `replica-identity.json` next to the DB. On restart,
  `ApplyPendingRestore` swaps the DB and the new
  **`db.ApplyReplicaIdentity`** rewrites this till's `sync.*` settings +
  receipt prefix into the restored data and clears the primary's
  sessions — so the replica is the shop's data with its own identity,
  not a byte-clone.
- **Wizard fork**: `/setup` gains "Join an existing shop" (step
  alternate; `POST /api/setup/join` is exempt but refuses once an
  operator exists). A factory-fresh till needs zero local setup — the
  snapshot brings catalog, settings, translations AND operators.
- **Per-till receipt sequences**: `NextReceiptNo` is prefix-aware
  (`sync.receipt_prefix`, e.g. `T2-`): each till counts within its own
  prefix, so numbers can never collide across tills (ADR-0011 rule).
  Primary stays unprefixed. Enrol response carries `till_no` (primary=1).

## Honest limits (documented for D3/D2b)

1. Plugin **files** don't travel with the snapshot — plugin rows do, so
   a replica shows the shop's plugins but must reinstall them from the
   marketplace (or wait for file sync). Item images likewise.
2. Post-join drift (catalog edits on the primary) isn't pulled yet —
   that's the D2b delta loop; D3 pushes replica sales back.
