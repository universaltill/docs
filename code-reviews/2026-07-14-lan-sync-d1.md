# Review — LAN sync D1: QR till pairing (ADR-0011, P2.3b)

Date: 2026-07-14 · Repo: universal-till · Spec: `architecture/lan-sync.md`
increment D1, per ADR-0011 (primary/replica).

## What shipped

- Migration 014 `tills` (primary side; per-replica bearer stored as
  SHA-256, sessions-style) + `TillsRepo` (insert/list/by-bearer-with-
  last-seen-touch/delete).
- **`/tills` page** (manager; linked from Settings): "Add a second till"
  → one-time pairing code (10-min expiry, in-memory store) rendered as a
  **QR** (go-qrcode, vendored pure-Go) + copyable JSON for manual entry;
  enrolled-tills table with last-seen and **Revoke**. The same page
  carries the replica-side "**Join an existing shop**" form (paste/scan
  the code → the till enrols itself against the primary and stores
  `sync.primary_url/till_id/bearer`).
- Sync surface: `POST /api/sync/enroll` (token-authed, middleware-exempt)
  → till row + bearer; `GET /api/sync/ping` (bearer-authed,
  middleware-exempt — the machine surface D2/D3 mounts alongside);
  `enroll-token`/`join`/`revoke` are manager-session endpoints. All
  enrolment events audited. 20 i18n keys en+fa.

## Notes

- Wizard "join" fork deliberately deferred to D2: without the snapshot +
  user sync, a joined fresh till has no operators to log in with. The
  Settings-page join covers D1's mechanics honestly.
- Enrol tokens are in-memory: a primary restart just means re-showing
  the QR.

## Verification

Unit suite + guards green. **Two-process E2E** (primary :8099 with auth,
replica :8098): pairing code issued (QR PNG data-URI + JSON) → replica
`join` → "✓ Joined: My Store", till row on the primary, `sync.*` settings
on the replica → bearer ping returns till id → wrong bearer 401 →
**token reuse 403 (one-time proven)** → revoke 204 → same bearer now 401.
