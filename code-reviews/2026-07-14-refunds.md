# Review — refunds & returns + receipt barcode (P1.0, G27/G28)

Date: 2026-07-14 · Repo: universal-till · Spec: `architecture/refunds.md`
(written first). Farshid: "I cannot see any refund!!!" — the engine could
do returns; no UI existed.

## What shipped

- **Receipt barcode (G28)**: `internal/print` renders CODE128 (`GS k` 73,
  code-set B, HRI below) of the receipt number on every sale AND refund
  receipt; non-ASCII/oversize codes are skipped rather than jamming the
  printer. The scan handler's last fallback: an unmatched code that is an
  existing receipt number → `HX-Redirect` to `/refund/{receipt}` — the
  cashier literally scans the paper receipt to start a return.
- **`/refund/{receipt}`**: original lines with per-line quantity pickers
  (decimals for weighed items) capped at the **remaining** refundable
  quantity (sold − already returned, summed over `sale_links` children,
  keyed item+variant+unit-price); refund method = cash or the original
  sale's methods; manager PIN field (skipped under `UT_AUTH=off`).
- **`POST /api/refund`**: `AuthorizeManager` (shared lockout; PIN owner =
  audit actor) → server-side double-refund guard (409 with a friendly
  message) → line + whole-sale discounts **prorated** by refunded share →
  tax mode inferred from the original sale's own header signature (not
  the current setting — it may have changed since) → `pos.CompleteSale`
  `SaleType:"return"` with `OriginalSaleID` (restocks, links) → audited →
  refund receipt auto-prints: `*** REFUND ***`, "For receipt <orig>",
  barcode, **drawer kick on cash** (giving money back).
- Journal: Refund button on sales; cross-links both directions
  (sale → "Refunds issued", return → "Refund of"). 11 i18n keys en+fa.

## Notes / accepted limits (spec'd)

1. Card refunds record the method; the merchant actions the money on
   their standalone terminal until G23 real payment plugins land.
2. Exchanges, restocking fees, no-receipt refunds: out of scope v1
   (journal search covers no-receipt lookups).
3. Refund total recomputed by mirroring the engine's math
   (`computeRefundTotal`) so the payment covers exactly — if engine total
   math ever changes, this helper must follow (single known duplication,
   commented).
