# Review — LAN sync D3: sale journal push (ADR-0011)

Date: 2026-07-14 · Repo: universal-till · Spec: `architecture/lan-sync.md`
increment D3. With this, a two-till shop sells anywhere and the primary
holds the consolidated truth.

## What shipped

- Migration 015 `sales.till_id` (provenance: '' = local, else the source
  till). `POSRepo`: `SaleExists`, `SetSaleProvenance` (till + original
  timestamp — CompleteSale writes "now"), `LocalSalesSince` (own sales
  past the cursor, journaled-in excluded), `OriginalSaleIDFor`;
  `SaleDetail` gains `CashierID`.
- **Journal = the sale detail; apply = `pos.CompleteSale` with the
  original ids/receipt** — the same engine produces the same stock
  movements, discounts, links and audit on the primary, and application
  is **idempotent by sale id**. Returns re-check the refund guard
  context; `AllowNegativeInventory` forced (the remote sale already
  happened — the books must reflect reality).
- Primary: `POST /api/sync/sales` (per-till bearer, batch ≤100, audited
  `sales_synced`). Replica: `StartSyncPush` 30s loop — push local sales
  past `sync.push_cursor`, advance on ack; primary unreachable = wait for
  the next tick (checkout never depends on it). Cursor is set to the
  join moment by `ApplyReplicaIdentity` (snapshot history must not
  re-push — though if it does, idempotency eats it, see below).

## Verification

Full suite + guards green. Two-process E2E with the D2-joined replica —
which had **no cursor** (joined before the cursor code), making it a live
idempotency torture test: the first tick pushed **15 sales** (its 2 own +
13 snapshot-history) and the primary answered `applied:2, skipped:13` —
no duplicates (16 rows, 16 distinct ids). The replica's `T2-000000001`
and a fresh `T2-000000002` (2× item) landed on the primary **stamped with
the till id**, stock consolidated (coke 29→28, pepsi −2), and the second
tick pushed nothing (cursor advanced). Shop-wide reports/Z-report now
include replica sales automatically (same tables).
