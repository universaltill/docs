# Code review — LAN sync D2b (admin drift pull) + D4 (status chips/UX)

Date: 2026-07-14 · Repo: universal-till · Branch: feat/lan-sync-d2b-d4
Spec: docs/architecture/lan-sync.md (ADR-0011 primary/replica)

## What shipped

**D2b — catalog/settings drift pull.** The primary serves its whole
admin-managed state as one bundle: `GET /api/sync/admin` (per-till bearer,
middleware-exempt like the rest of `/api/sync/*`). Replicas poll every 30s
with `?have=<fingerprint>`; a matching SHA-256 fingerprint answers
`unchanged` with no apply. On drift the replica applies the bundle in one
transaction — **primary wins**, deletes propagate.

- `internal/data/sync_admin_repo.go` — `DumpAdmin`/`ApplyAdmin` +
  `AdminBundle.Fingerprint()`. Synced tables (FK-safe order): tax_codes,
  brands, categories, customers, payment_methods, users, items,
  item_barcodes, item_variants, variant_barcodes, related_items,
  promotions, shortcut_buttons, translation_overrides, settings.
  Deliberately NOT synced: inventory/stock (additive movements are D3's
  job), sales, sessions, stock_locations, item_images (files don't
  travel), plugin tables (replicas install from the marketplace).
- Apply = **prune-then-upsert**: deletes run first (children first) so a
  replica-local row squatting on a UNIQUE value (e.g. same SKU under a
  different id) clears before the primary's row lands. A prune blocked by
  FKs (row referenced by local sales history) falls back to
  `is_active = 0` where the table has it, else logs and keeps the row.
- **Per-till settings never travel**: `sync.*`, `printer.*`, `display.*`,
  `reports.eod_*` (`data.PerTillSettingPrefixes`) are filtered from the
  dump AND shielded from prune/upsert on the replica (defense in depth).
  `reports.eod_*` stays per-till because a replica Z-report would cover
  only its local slice.
- After an apply the replica re-derives in-memory state exactly like the
  settings handlers do on a manual edit: theme/tax/currency via
  `LoadState`→`UpdateState`, `httpx.InitCurrency`, a fresh `pos.Service`,
  idle-lock minutes, and shop translation overrides into the localizer —
  verified live: an override set on the primary rendered on the replica
  without a restart.

**D4 — status chips + banner + promote doc.**

- `/ui/sync-chip` polled by the nav every 30s. Replica: till name +
  push-queue depth (`POSRepo.CountLocalSalesSince`), amber ⚠ when the
  primary hasn't answered for 90s (both loops stamp
  `sync.last_contact_at`). Primary: enrolled-till count, amber when any
  till unseen for 2 minutes, links to /tills. Renders empty on a
  standalone till — feature invisible until used.
- Replica catalog page gets a banner ("this till follows the primary…")
  linking to the primary's catalog. **Spec deviation, recorded in
  lan-sync.md:** no redirect/queue-and-warn — a redirect needs a
  reachability probe in the page path and queued edits need a conflict
  story; the 30s primary-wins pull already resolves local edits honestly.
- Manual promote-replica procedure documented in lan-sync.md.

## Design decision (deviation from spec wording)

The spec sketched per-table "changed since cursor" deltas. Most admin
tables have no `updated_at` (categories, users, barcodes…), and cursors
can't express deletions. A whole-bundle fingerprint costs a full read per
poll (fine for shop-sized SQLite on LAN) and buys correct deletes, no new
migrations, and no touch-on-write bookkeeping. lan-sync.md updated.

## Review findings — multi-angle review (3 finder agents), all fixed

**Fixed before merge:**

1. **Checkout could fail with SQLITE_BUSY during an apply** (CONFIRMED).
   `db.Open`'s DSN used mattn-style `_foreign_keys=on` — silently ignored
   by modernc — and the Exec'd `PRAGMA foreign_keys` reaches only ONE
   pooled connection, so **FK enforcement was missing on pooled
   connections (pre-existing bug)** and there was no busy handler at all:
   a sale INSERT during ApplyAdmin's write transaction would error
   "database is locked". Fixed: DSN now
   `_pragma=foreign_keys(1)&_pragma=busy_timeout(5000)` (modernc syntax,
   applies to every pooled connection).
2. **Refresh closure wiped an in-flight basket** (CONFIRMED). It replaced
   `dp.Engine` — the singleton holding the current basket — with a fresh
   empty service on every applied pull; a catalog edit on the primary
   would silently clear a cashier's half-scanned sale on the replica.
   Fixed: new `pos.Service.SetConfig` swaps tax config **in place**
   (basket preserved, totals recomputed), called only when the config
   actually changed. The settings handlers still use the replace pattern
   (operator-initiated, rare) — follow-up to migrate them to SetConfig.
3. **Prune fallback could poison every future pull** (CONFIRMED). A
   replica-local row that is FK-pinned by local history AND holds a
   UNIQUE-but-not-PK value the primary now uses (items.sku,
   users.username, …) was deactivated but kept the value → the upsert
   UNIQUE-violated → the whole apply rolled back, **on every pull,
   forever**. Fixed: retiring a row also mangles its UNIQUE columns
   (`COLA` → `COLA~itm-local`, idempotent CASE guard). Regression test
   applies the same bundle twice.
4. **Settings prune was a version-skew footgun** (PLAUSIBLE). A
   non-per-till key a newer replica writes that the primary doesn't have
   would be deleted on every pull. The app only ever upserts settings —
   pruning them bought nothing. Fixed: settings are never pruned.
5. **Cleanups** from the same review: shared `runSyncLoop` harness for
   push+pull ticks; chip handler reuses `Deps.SyncPrimaryURL` and hoisted
   repos; `Deps.SyncPrimaryURL` made nil-store-safe (catalog handler
   tests build bare Deps).

**Found while building (fixed pre-review):** prune-before-upsert ordering
for the simple SKU-squatter case; per-till settings shielded on the apply
side too (a buggy primary must not push `sync.bearer`); UI scale no longer
zeroed by the refresh when `display.ui_scale` is unset.

**Known/accepted (documented, not fixed):**
- The primary re-dumps all admin tables per poll (~every 30s per
  replica). Fine for shop-sized SQLite on LAN; add a write-invalidated
  fingerprint cache if it ever shows up in profiles.
- `users` sync carries `pin_hash` to enrolled tills (bearer-gated; the D2
  snapshot already ships the whole DB) — deliberate, so shop PINs work on
  every till. Deactivating a user on the primary logs them out of
  replicas within a tick (sessions check `is_active` — desired).
- A held sale referencing an item the primary hard-deleted keeps working
  from its JSON snapshot, but completing it would hit the items FK. The
  app never hard-deletes items (deactivate only), so this needs a
  hand-edited primary DB.
- Version-skew NOT NULL risk on upsert is theoretical: SQLite's
  `ALTER TABLE ADD COLUMN NOT NULL` requires a DEFAULT, and migrations
  are append-only ALTERs after v1.
- `/api/sync/admin` error path uses `http.Error` not the JSON envelope —
  matches the pre-existing pattern on the sibling sync endpoints.

## E2E (two processes, one machine)

Primary :18080, replica :18081 (UT_AUTH=off, separate DBs), real QR
pairing code → join → restart. Then, all through the real APIs:
item rename+price change, new barcode, `shop.name` — all on the replica
within one tick; replica's `printer.host` and `sync.*` untouched; barcode
delete propagated; translation override rendered live on the replica
("Sell Screen" in nav, no restart); replica sale showed
"⇅ Till 2 · 1 queued" then drained to the primary (T2-000000001, till
provenance stamped); 4 applies over ~20 polls = unchanged short-circuit
works; killing the primary flipped the replica chip amber; primary chip
shows "⇅ 1 tills". Checkout never touched the network (ADR-0003).
